JSNaughty

JSNaughty attempts to reverse engineer identifier obfuscation in Javascript using statistical machine translation (specifically Moses). The goal is to take obsfucated idenfiers from minified Javascript, and rename them to be descriptive in reading the code. In its original context, Moses is used to translate one human language to another, and there are both benefits and challenges in adapting it to an identifer renaming context. The major benefit to using it on minifed Javascript is that we are only translating the identifier names. The context remains the same in each "language", so the scope of the task is much smaller than in translation. However, the major challenge is that the renaming must be consistent - variables on different lines must use the same name in the translation, if they are in the same scope. Likewise, different variables in the same scope must remain different, and both of these issues are not resolved within Moses. Therefore, we use a series of techniques in the pre and post-processing to insure that naming in the translation is consistent and preserves the semantics of the original minified code.

Natural Language and Programming Language Comparisons

Intuition suggests people find reading and writing programs more difficult than their native languages. However, statistical langauge modelling shows that the actual text of natural language is less repetitive, less predictable, and more ambiguous than code. Some of this restrictiveness is imposed by the constraints of code - the need for code to compile, restricted grammar, etc, but is some of it also a delibrate choice by developers to constrain their choices in code because the language is more difficult? While good coding style recommends code re-use via modularization, copy pasted code and patterns are still quite common. I would like to more formally define language difficulty and associate it with the observable features of humans. This would allow a grounded approach to comparing not only between natural language and programming language, but also between different programming languages. Knowing what elements of programming are most cognitively difficult would allow better tools and more targetted approaches to making programming languages both easier to learn and use.

Automatic Syntax Correction in Student Programs

Finding and fixing software defects are very costly, causing billions of dollars of expenses each year. Many techniques to reduce this cost or prevent defects have been developed, but the problem persists. One tactic to resolve this problem is to automatically find and fix errors. However, due to the scale of the problem, this is no easy task, and it is likely that only a subset of defects will be amenable to automatic fixes.

Therefore, we have developed a more controlled dataset of simple student programs to test using language models and machine learning techniques. We track student submissions through version control to see how they develop over time, gaining information on how many students attempt the sample assignments, the kinds of defects introduced, and the fixes developed to create correctly working programs. The goal is to find what machine learning and language modeling techniques can best learn from the student fixes and how effective they can be. If we can find effective enough fixes in this much more controlled context, we would like to extend what we find to larger programs to see what bugs these techniques can fix in a real software context.

Gitcproc

Gitcproc is a tool to parse git logs to discover the context in which code structures appear in a language independent manner. The idea is to use regular expressions to capture function context and scope changes to track where user defined terms change in a more reliable way that the basic git suite. Currently works for Python, C, C++, and Java.
